# 执行模式判定（交互式 vs 非交互式）

> SKILL.md 在「输入采集流程」前必须先完成本节判定。判定结果决定后续是否允许调用 `AskUserQuestion`。

## 判定流程

1. **解析原始 query**，按下表抽取字段，构造字段表 `parsed`：

| 字段 | 数据来源关键词示例 | 解析结果 |
|------|------------------|---------|
| `destination` | "去日本东京"、"丽水"、"悉尼+墨尔本" | 国家 + 城市数组（≥1） |
| `dates` | "6/28-6/30"、"7/8 出发"、"2026-07-01 到 07-05" | 精确出发日期 + 返回日期 |
| `days` | 由 `dates` 推出，或显式 "玩 10 天" | 整数天数 |
| `people` | "我和老婆带 8 岁孩子"、"2 大 1 小"、"出差一个人" | 成人/儿童/老人数 + 儿童年龄段 |
| `transport` | "自驾"、"高铁"、"飞过去" | 至少 1 项 |
| `trip_type` | "亲子游"、"商务"、"摄影之旅"、"潜水" | 至少 1 项 |
| `extras` | "无人机"、"露营装备"、"宠物"、"婴儿" | 0~N 项 |

2. **触发非交互模式**（满足以下任一即触发）：
   - **A. 显式标记**：原始 query 包含 `[non-interactive]`、`[auto]`、`--auto`、`--non-interactive` 任一标记（大小写不敏感）。
   - **B. 字段完整度**：`parsed` 中 `destination`、`dates`、`days`、`people`、`transport` **全部** 非空（5 项核心字段齐全），且不存在歧义日期（"7 月"、"暑假"、"下周"等模糊时间不算 `dates` 命中）。
   - 任一命中 → 进入【非交互执行路径】。

3. **未命中** → 进入【交互执行路径】，按 SKILL.md 「输入采集流程」走 AskUserQuestion。

## 非交互执行路径（禁用 AskUserQuestion，但 special_needs 是唯一例外）

- **硬约束**：本路径下**禁止调用 `AskUserQuestion`**——**仅有 1 个例外**（见下方「special_needs 强制确认门」）。所有其他缺字段一律用默认值填补，不发问、不阻塞。

### special_needs 强制确认门（非交互路径下唯一允许发问的字段）

**为什么必须开这个例外**：
- `destination / dates / days / people / transport / trip_type` 六项都能从 query 关键词反推（"自驾"→transport、"和老公"→people、"美西"→destination 等）。
- **`special_needs`（无人机 / 户外露营 / 额外医疗用品）是唯一无法反推的字段**——用户不主动声明，AI 猜不出，也不能默认为"无"：
  - 无人机需要额外装备（备用电池 ×2 + 桨叶 + ND 滤镜 + 目的地法规查询），漏掉一整卡装备
  - 户外露营需要帐篷/睡袋/防潮垫/头灯/炊具，漏掉一整卡装备
  - 慢性病处方药需要英文说明 + 加大用量，漏掉可能影响健康
- **"字段有默认值"≠"默认值一定正确"**：`special_needs=无` 在多数场景下都错（尤其自驾/户外/摄影/长途行程）。

**执行规则**：
- 即使命中非交互路径（显式标记 or 五字段完整），**仍必须在开始生成清单前发一次 AskUserQuestion 单题**：
  - question："这趟有没有需要额外准备的特殊装备？"
  - multiSelect: true，4 选项：无人机航拍 / 户外露营/星空 / 额外医疗用品 / 无额外需求
  - 选项文案按目的地和出行类型微调（如高原行程可换"高原氧气瓶"、海岛行程可换"潜水装备"）
- **query 关键词嗅探优先**：若原始 query 已含"无人机/航拍/drone""露营/camping/星空/帐篷""处方药/慢性病/医疗"任一关键词，视为用户主动声明，`extras` 已知，**跳过本次确认**不发问。
- **显式标记豁免**：若 query 含 `[non-interactive]` / `[auto]` / `--auto` / `--non-interactive` 任一，说明调用方明确要求全自动，此时 `special_needs` 默认为"无额外需求"，也不发问。
- **发问位置**：所有其他字段确认后、天气 API 调用前——一次只发这一题，用户 3 秒点完，随后进入清单生成。
- **违反后果**：跳过本确认门直接生成清单，等同于承担"漏掉一整卡装备"的风险，是本 skill 上一版最高频的用户投诉根因。

### 其他缺字段默认值（可静默填补，不发问）

| 字段 | 默认值 | 说明 |
|------|-------|-----|
| `dates` | 今日为锚点，且置 `approximate=true` | 不再硬编"今天+7天"。后续天气走 L3 气候常态（`references/weather-api.md` L3），HTML 头部标注"出发日未定" |
| `days` | 5 | 折中常见值 |
| `people` | 2 个成人 | 不假设儿童/老人 |
| `transport` | 国际 → 飞机；国内跨城 → **不设默认值，必须追问**（追问时 transport 视为缺字段，走 special_needs 确认门同级的单题 AskUserQuestion）；短途/同城 → 其他 | 国内跨城飞机与高铁在液体限制、托运规则上差异大 |
| `trip_type` | 休闲度假 | 通用兜底 |
| `extras` | **禁止默认为空——必须通过上方「special_needs 强制确认门」获取**（query 嗅探命中 or 单题确认 or 显式标记豁免三选一） | 无法从 query 反推的字段，不能静默默认 |
| 多城市天数划分 | 平均分配（不能整除时余数给首城市） | 等同「快速版」语义，不再追问 |
| 儿童年龄段（含儿童时） | 4-7 岁（学龄前） | 中位段位 |

- **回显声明**：在生成清单前，**首条文本**输出一段简短「执行参数确认」清单（destination / dates / days / people / transport / trip_type / extras），并明确标注哪些字段来自 query、哪些来自默认值（标注 `[default]`）、哪些来自 special_needs 确认门（标注 `[confirmed]`），便于真人或评测者复核。
- **自检表豁免**：SKILL.md 「自检表」中要求"停止生成、立即追问"的项，在非交互模式下改为"使用默认值并在执行参数确认中标注 `[default]`"；自检字段表本身仍需逐项打勾，缺一不可。
- **天气 API**：仍正常调用；若失败走原有 fallback（气候平均值），不发问。

## 交互执行路径（默认）

按 SKILL.md 「输入采集流程」走 AskUserQuestion，与原行为一致。注意：「已知信息跳过提问」依然生效——`parsed` 中已有的字段不再追问，只补缺。
