# 多城市行程文件分支（SKILL.md 第 2 节详细版）

> 本文件是 SKILL.md「2. 多城市行程文件分支」的完整展开版。
> SKILL.md 主文件仅保留触发条件与行为概要，进入多城市行程处理阶段时按需加载本文件获取完整流程。
>
> 加载时机：Step 3 城市追问结果中用户选择了 ≥ 2 个城市时，读取本文件执行完整的多城市文件分支流程。

---

## 触发条件

- Step 3 城市追问结果中用户选择了 ≥ 2 个城市（如云南→昆明+大理+丽江+香格里拉）

## 行为

### 0. 前置检查（上下文感知，禁止重复提问）

进入多城市分支后，**先检查 Step 1 的采集结果**，再决定走哪条路径：

| Step 1 用户选择 | 多城市分支行为 |
|----------------|--------------|
| 「上传/拖拽行程文件」或「直接粘贴」且**解析成功** | **跳过整个多城市分支**——文件解析已提供城市顺序和天数，直接进入 Step 4 剩余字段（人员/交通/特殊需求） |
| 「上传/拖拽行程文件」或「直接粘贴」但**解析失败且尚未回退** | 走下方 Section 1 文件分支询问（给用户一次重试机会） |
| 「上传/拖拽行程文件」或「直接粘贴」但**解析失败且已回退到手动输入** | **等同于「手动输入」**——跳过下方文件询问，直接进入 2.1 精确度选择 |
| **「没有现成的，手动输入」** | **跳过下方「文件上传/粘贴」询问**，直接进入 2.1 精确度选择（快速版/精确版） |

> ⚠️ **硬性规则**：整个会话中，"有没有行程文件"这个问题**最多出现 1 次**（在 Step 1）。多城市分支不得再次询问同一问题。文件解析失败并已回退到手动输入的，视为用户已明确表态"没有行程文件"，不得重试。

### 1. 文件分支询问（仅当 Step 1 未走过手动输入路径时执行）

在 Step 4（交通方式 / 特殊需求）之前，**先**插入一个轻量分支问题，让用户选择走"文件直读"还是"结构化问答"。**提示语必须显式提及"每个城市待几天"**，让用户提前知晓多城市行程需要天数分配信息：

> 提示语示例："这趟行程涉及多个城市，要做到逐天穿搭建议需要知道**每个城市各待几天**。如果你已经有现成的行程规划，直接丢一份给我能省去几轮提问。"
>
> 选项（单选）：
> - 上传 / 拖拽行程文件（支持 .md / .txt / .docx / .pdf / 截图）
> - 直接粘贴一段文字行程到对话框
> - 没有现成的，按问答继续（会追问每个城市的天数分配）

走前两个分支 → 进入「文件解析子流程」；走第三个分支 → 进入「精确度选择」子流程（见下方 2.1）。

### 2.1 精确度选择（快速版 vs 精确版）

**触发时机**：用户在 2 分支问中选择了「没有现成的，按问答继续」时，**立即**追加一个精确度选择问题（与后续 Step 4 合并为同一次 AskUserQuestion 亦可）：

> 提示语示例："多城市行程如果要做到逐城逐天的穿搭建议，需要你告诉我每个城市待几天。如果你还没想好，也可以先出一份通用版清单。"
>
> 选项（单选）：
> - 快速版：按整体温度范围出清单，不细分每城天数（少问几个问题，更快出结果）
> - 精确版：我来分配每个城市的天数，生成逐城逐天穿搭建议

**快速版行为**：
- **跳过** Step 4 中的「多城市天数划分」问题（如适用）
- 天气数据：对所有目的地城市统一查询行程日期范围内的温度，取所有城市的温度极值范围（最低温 ~ 最高温）作为整体衣物推荐依据
- 按天穿搭建议：**不按城市分段**，改为输出"整体温度范围 + 分层穿搭建议"的简化版（如"行程期间各城市温度 15-38°C，建议洋葱式穿搭：短袖为主 + 薄外套应对空调/早晚"），不逐天列出
- 自检表中「各城市天数」字段标记为"快速版豁免"，不阻断生成
- 其余采集字段（交通方式、人员构成、出行类型、特殊需求）仍在后续步骤正常采集

**精确版行为**：
- 与标准流程一致，Step 4 正常插入「多城市天数划分」问题
- 按天穿搭按城市分段逐天输出

### 2.2 文件解析子流程

**支持格式与读取工具**：

| 格式 | 工具 |
|------|------|
| `.md` / `.txt` / 粘贴文本 | Read |
| `.docx` | docx skill |
| `.pdf` | pdf skill |
| 图片截图（小红书 / 备忘录截图等） | 多模态读图（直接 Read 图片） |

**必须抽取的字段**（与自检表对齐）：

| 字段 | 抽取目标 | 缺失/冲突时降级 |
|------|---------|---------------|
| 出发 / 返回日期 | `YYYY-MM-DD` 或 `M/D` | 退回 AskUserQuestion，仅追问日期 |
| 城市顺序 | `[城市A, 城市B, …]`，按 Day 顺序 | 用文档出现顺序，回显时让用户确认 |
| 各城天数 | `{城市: 天数}`，Σ天数 = 总天数 | Σ 不一致时强制追问 |
| 城际交通 | 飞机 / 高铁 / 自驾 / 大巴 | 缺失时合并到 Step 4「交通方式」题 |
| 关键活动关键词 | 按 Day 标注的活动短语 | 选填，仅用于装备反推（见 packing-rules.md 第 13 节） |
| 住宿点 | 选填 | 缺失忽略 |

**隐私脱敏**：识别到手机号、身份证号、护照号、订单号、银行卡号等敏感字段时，**不得**写入回显卡片，仅在末尾提示"已忽略个人敏感信息"。

### 2.3 回显确认（置信度判断，条件性触发）

抽取完成后，**先进行置信度判断**，决定是否需要回显确认：

**高置信度（跳过回显）**——同时满足以下全部条件时，直接进入 Step 4 补缺，不弹确认卡片：

1. 核心字段完整：出发日期、返回日期/总天数、城市顺序均已抽取且无歧义
2. 各城天数之和 = 总天数（±1天容差，含到达/离开日）
3. 日期为精确日期（非"月中""下旬"等模糊表述）
4. 无字段间冲突（如文本说"5天"但逐城天数加起来是7天）

> 跳过回显时，在进入 Step 4 补缺提问前，用一行自然语言简要确认解析结果（如"行程已识别：7/20-27 东京3晚→京都2晚→大阪2晚，接下来补几个问题～"），但**不使用 AskUserQuestion**，不阻断流程。

**低置信度（触发回显确认）**——命中以下任一条件时，必须回显并等待用户确认：

- 核心字段有缺失（如缺日期、缺城市）
- 各城天数之和 ≠ 总天数
- 日期存在歧义（如"下周五"需推算）
- 字段间有冲突
- 文本噪声较多（如小红书攻略截图含大量无关内容）

低置信度时用结构化卡片回显：

```
 已从你的行程文件中识别：
- 总日期：YYYY-MM-DD ~ YYYY-MM-DD（共 N 天）
- 城市顺序：A → B → C
- 各城天数：A 3 / B 4 / C 3
- 城际交通：A→B 高铁 / B→C 飞机
- 主要活动：富士山徒步、环球影城、温泉
（已忽略 X 项个人敏感信息）
```

随即用 AskUserQuestion 让用户单选确认：

- ✅ 信息正确，按这个生成
- ✏️ 我要改一两项（用户选此项后再发起追问，仅针对要改的字段）
- 🔄 文件解析有误，改用问答（退回原结构化流程）

**低置信度路径下严禁**在用户未点击确认前调用天气 API 或生成清单。

### 2.4 与原流程的衔接

- 已抽取并确认的字段视为已知，按「3. 已知信息跳过提问」表格的标准跳过对应问题
- 后续步骤（Step 4）仅就**仍然缺失**的字段（人员构成 / 儿童年龄 / 交通方式 / 特殊需求）发起提问
- 多城市天数划分题：若文件解析回显已确认，**直接跳过**该题
- 若用户在 2.3 选择「🔄 改用问答」，本次会话不再尝试文件解析分支，直接走 Step 2 手动输入流程

### 2.5 异常处理

- **30 秒内未收到附件**：主动追问"是没找到文件，还是想直接打字？"，给一条「改用问答」退路
- **文件过短或解析失败**：给出失败原因（如"未识别到日期 / Day 标记"），让用户在「重传 / 改用粘贴 / 改用问答」三选一
- **行程粒度过粗**（如只有"东京 3 天，京都 2 天"）：等价于结构化输入，**简化回显**（仅用一行文字确认如"已识别：东京3天 + 京都2天，共5天，对吗？"），不再输出完整结构化卡片。用户确认后字段直接填到采集结果，进入剩余字段提问
- **截图含大量噪声**（小红书攻略截图常见）：解析时只保留含日期 / Day 标记 / 城市名的段落，丢弃营销文案
- **解析连续失败 2 次**：自动退回 Step 2 手动输入路径，并提示"已切换到问答模式"
