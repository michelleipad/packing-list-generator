# 行李清单生成器

## 触发边界（快速校验）

本 skill 仅处理"出行/旅行/出差时带什么东西"的场景。

**不触发**：含"采购/装修/进货/备货/购物/搬家/年货/婚礼/月子/食材/物料/施工/工程"等非出行限定词时，即使同时出现"清单""帮我列"也不触发——直接回复"我只帮整理旅行行李"，不调用任何工具、不生成文件。

**也不触发**：行李箱故障维修、纯签证/机票/酒店咨询（未提物品）、物品选购建议、已生成清单后的单项增删。

## 已知信息跳过提问

**HARD GATE（进入本 skill 前的调用方也必须遵守）**：任何情况下，都必须先把用户原始 query + 上游 AskUserQuestion 已回答的字段（含 ARGUMENTS 传入的 K:V）**逐字段登记到 `known_fields`**，登记后**永久锁定**，本 skill 内后续所有环节禁止再对同名字段发问。

登记规则（明确/近似都算已知，不再追问）：
- 目的地：query 或 ARGUMENTS 含具体国家/城市 → `destination` 已知
- 出发日期：精确日期 → `dates` 已知；仅月份/模糊时间且用户已在上游轮次表达过"就本月/暑假/大概那时候" → 直接 `approximate=true` + 该月 15 号锚点，**不再追问精确日期**；**query 完全未提任何时间线索（无月份、无季节、无"暑假/年假/国庆"等假期词、无"下周/下月"等相对时间）→ `dates` 状态为 UNKNOWN，必发 Step 2 第 3 题追问，禁止根据目的地典型旺季自行推断月份**（如"新疆/西藏→暑期""东北/北海道→冬季""日本→樱花季"这类目的地-季节联想是高频误判源，任何情况下都不构成上游线索）
- 天数：任何精确数字（含"5天""7 days""半个月=15"）→ `days` 已知，**禁止再追问"具体几天"**；仅模糊词（十来天/一周多）→ 单次追问一次，用户拒绝立即 approximate_days=true + days=7
- 出行类型：query 含"自由行/观光/购物/亲子/商务/摄影/滑雪/潜水/徒步"等关键词 → `trip_type` 已知
- 人员构成：query/ARGUMENTS 明确 → 已知；**完全未提及 → 默认"1个成人（自己）"，不追问**
- 交通方式：跨国出行未指定 → 默认"飞机"，不追问；**国内跨城未指定 → 不设默认值，必须追问**（飞机与高铁在液体限制、行李托运、充电宝规则上差异大，默认高铁容易误导用户遗漏飞行限制项）；短途/同城未指定 → 默认"其他"，不追问
- 特殊需求：未提及 → 默认"无额外需求"，不追问

**冗余追问检测**：进入 Step 1-4 任何一步前，先扫描 `known_fields`，命中的字段整题跳过。若跳过后本 Step 全部字段都已知，直接跳到下一 Step。若 Step 1-4 全部字段都已知，直接进入天气 API + 清单生成，**一次 AskUserQuestion 都不发**。

**违规后果**：对已在 `known_fields` 中的字段再次发问，等同于让用户重答一遍上一轮已回答的问题，用户流失率极高——宁可用默认值生成后允许用户编辑，也不重复追问。

## 执行模式判定（交互式 vs 非交互式）

> **详细规则已外置** → `references/execution-mode.md`
>
> **核心原则**：先判定再采集。命中下列任一条件即进入【非交互执行路径】，**禁用 AskUserQuestion**，缺字段走默认值：
> - **显式标记**：query 含 `[non-interactive]`、`[auto]`、`--auto`、`--non-interactive` 任一（大小写不敏感）
> - **字段完整度**：解析后 `destination` / `dates` / `days` / `people` / `transport` 全部非空（且日期非歧义）
>
> 未命中则进入【交互执行路径】，按下方「输入采集流程」正常发问。
>
> ⚠ **非交互路径的唯一例外——special_needs 强制确认门**：即使命中非交互路径，`special_needs`（无人机 / 户外露营 / 额外医疗）是唯一无法从 query 反推的字段，AI 也不能默认为"无"——必须在天气 API 调用前发一次单题 AskUserQuestion（4 选项：无人机航拍 / 户外露营·星空 / 额外医疗用品 / 无额外需求）。**豁免条件**：① query 已含"无人机/航拍/露营/星空/处方药/慢性病"等关键词视为已声明；② query 含显式非交互标记（`[non-interactive]` 等）视为调用方要求全静默、默认"无"。详见 `references/execution-mode.md`「special_needs 强制确认门」。
>
> **读取时机**：进入输入采集前，若需要查表确定缺字段默认值或回显格式，读取 `references/execution-mode.md`。

## 输入采集流程

> 本节仅适用于「交互执行路径」。非交互路径请按 `references/execution-mode.md` 的默认值表生成，并在生成前回显「执行参数确认」（标注 `[default]`）。

分步骤完成，使用 AskUserQuestion 工具采集。

### 第一轮：基础框架（锁定"去哪 + 何时"，Step 2-3 完成目的地和日期采集）

**Step 1**：行程文件优先采集（减少手动输入轮次）

使用 AskUserQuestion 询问用户是否有现成行程文件：

> 提示语示例："如果你已经有行程规划，直接丢一份给我能省去几轮提问。"
>
> 选项（单选）：
> - 上传 / 拖拽行程文件（支持 .md / .txt / .docx / .pdf / 截图）
> - 直接粘贴一段文字行程到对话框
> - 没有现成的，手动输入

走前两个分支 → 进入「2.2 文件解析子流程」→ 2.3 回显确认 → 仅追问缺失字段（跳到 Step 4）；走第三个分支 → 进入 Step 2 手动输入。

**Step 2**（无行程文件时走此步）：使用 AskUserQuestion 一次性采集以下 4 个问题：

1. **目的地区域**（multiSelect: true，AskUserQuestion 每题最多 4 选项，按区域分组）
   - 选项：中国（国内）、东亚（日韩）、东南亚（泰新马越等）、欧美澳（美英法澳等）
   - 用户可选择"Other"输入具体国家名
   - 选完区域后在 Step 3 追问具体国家/城市

2. **出行类型**（multiSelect: true，最多 4 选项）
   - 选项：休闲度假（含蜜月/亲子游）、商务出差、户外运动（徒步/滑雪/潜水）、摄影/深度游
   - 用户可选择"Other"输入其他出行类型

3. **出发日期**（采集到精确日期再继续，因为天气 API 需要 `YYYY-MM-DD` 才能定位逐日预报）
   - ⚠ **前置守卫（HARD GATE）**：query **完全未提任何时间线索**（无月份、无季节、无"暑假/年假/国庆"等假期词、无"下周/下月"等相对时间）时，`dates` 状态为 UNKNOWN，**必发本题**。**严禁**根据目的地典型旺季自行推断月份——"新疆/西藏→暑期""东北/北海道→冬季""日本→樱花季""海南→冬季避寒"等目的地-季节联想是本 skill 最高频的误判源，任何情况下都不构成可用线索。宁可多问一次，也不要脑补日期让下游天气/衣物全盘错位。
   - 把日期采集到 `YYYY-MM-DD`（或 `M/D`）级别。若用户只给出"7月"、"下月"、"暑假"、"下周"等模糊时间，通过 AskUserQuestion 追问具体日期。
   - **选项设计（4 选项一步到位）**：3 个精确日期参考（如"本周末（{下一个周六的 M/D}）"、"下周一（{下一个周一的 M/D}）"、"下月 1 号（{下月 1 号的 M/D}）"）+ **第 4 个固定选项**："暂未确定，按近似日期估算"（含 Other 自定义入口）。
   - **降级规则（关键 ）**：若用户选择"暂未确定"或 Other 填入"不知道/没定/再说"等非日期内容，立即设置 `approximate=true`，不再二次追问——重复追问会让对模糊日期持开放态度的用户感到流程冗余而中途放弃。锚点日期取法：① 用户原 query 含月份 → 该月 15 号；② 仅给天数无月份 → 今日（now）。
   - ⚠ **天气链路硬绑定（HARD GATE）**：`approximate=true` 时**禁止**调用 forecast API——锚点日期（如 7 月 15 号）极大概率落在 forecast 的 `today ~ today+16` 窗口之外，forecast 会整段返回 400 而不是部分裁剪，导致流程中断。此时**必须直接走 `references/weather-api.md` L3 气候常态分支**（climate-api 或 WebSearch 月度均值），锚点日期**仅用于 HTML 头部 `.approx-notice` 文案生成**，不作为任何 API 的 `start_date`/`end_date` 参数。
   - 不要在用户仅说月份时自行假设具体某一天（如默认月中、月初、本月某周五）；隐式假设的日期会让生成的天气数据偏差 ±15 天，进而推荐错季节衣物。仅当用户**显式选择"暂未确定"**后，才允许使用月内 15 号作为锚点，并在 HTML 头部标注"出发日未定，按 {目的地} {月份} 气候常态估算"。

4. **行程天数**（选项给精确数字，因为衣物件数 ceil(天数/频率) 直接依赖天数）
   - **选项设计（4 选项一步到位）**：3 个精确天数（5天、7天、14天）+ **第 4 个固定选项**："暂未确定，按 7 天估算"（含 Other 自定义入口）。用户可在 Other 中输入任意精确天数。
   - 用户选完精确数字直接采用，不要再追问"具体几天"——重复追问会让用户感到流程冗余。
   - **降级规则（关键）**：若用户选择"暂未确定，按 7 天估算"，或 Other 填入"不知道/没定/再说/待定/不确定/随便/看情况"等非数字模糊内容，立即设置 `approximate_days=true`、`days=7`，不再二次追问——重复追问会让对天数持开放态度的用户（出差/Gap 行/签证待批等场景）中途放弃。锚点理由：7 天是国内长假典型时长，且 ceil(7/频率) 在常见频率（2/3/5 天换洗）下都能给出合理件数；用户拿到清单后可在 HTML 上手动加减。
   - 不要出现"10天以上""一周左右"等模糊档位作为精确选项，因为模糊档位无法推算返回日期，也无法用于按天穿搭分段。降级只通过显式的第 4 选项触发，不要在用户输入"7天"等精确值时自作主张降级。
   - 出发日期 + 行程天数 = 行程日期范围，用于调用天气 API 获取逐日预报；`approximate_days=true` 时仍按 7 天范围拉天气数据，但 HTML 渲染改用两段式穿搭（详见输出规范）。

**Step 3**：用户选完国家后，**立即**发起 AskUserQuestion，针对每个已选国家追问城市（每个国家一个问题，multiSelect: true）：

- 每个国家最多提供 4 个热门城市选项（如美国：纽约、洛杉矶、旧金山、芝加哥），用户可通过"Other"输入其他城市
- 小国用首都或主要城市（如新加坡：新加坡；韩国：首尔、釜山、济州岛）
- **非热门国家兜底**：当目的地国家不在常见旅游热门列表中（如塞内加尔、乌兹别克斯坦、玻利维亚等），AI 无法自信列出该国 4 个热门城市时，**禁止猜测首都作为默认值**——直接用一个开放式问题让用户自填城市（选项设为 1-2 个该国仅有的知名城市 + 留足 Other 入口），避免因默认首都而遗漏用户实际要去的城市的天气和装备需求
- 如果只选了"中国（国内）"，同样追问具体城市（北京、上海、广州、成都等，最多 4 个选项）
- 最多支持 4 个国家（AskUserQuestion 上限），超过 4 个时拆分提问
- **多城市触发**：若 Step 3 结果中用户选择了 ≥ 2 个城市，先走「2. 多城市行程文件分支」再进入 Step 4——直接跳进 Step 4 会丢失分城市天数划分，按天穿搭无法分段渲染。
  - 📄 **详细分支细则** → `references/multi-city-file-branch.md`（含文件解析置信度判定、回显确认门槛、快速版/精确版切换、天数划分兜底策略）。**读取时机**：走到多城市分支且需要展开细则时加载。
  - ⚠️ **上下文感知（避免重复提问）**：若 Step 1 用户已选择「没有现成的，手动输入」，则多城市分支中跳过文件上传/粘贴询问，直接进入 2.1 精确度选择（快速版/精确版）。再次询问"有没有行程文件"会让用户觉得 AI 没在听上一轮的回答。
  - 若 Step 1 用户选择了上传/粘贴文件并成功解析，则多城市分支中跳过精确度选择和天数划分（已从文件获取），直接进入 Step 4 剩余字段。
> ⚠️ **国际出行提醒（HARD GATE）**：当目的地为任何非"中国（国内）"的国家时，本轮回复末尾追加一句国际出行提醒，内容须明确提及"护照"或"签证"至少一个关键词。示例："去日本记得提前确认护照有效期和签证哦～"——遗漏这句提醒会让用户在临走前才发现证件没办，是本 skill 国际场景下最高频的事故根因。

**Step 4**：城市确认后，发起 AskUserQuestion 采集以下条件字段。

> **批次策略**：Step 4 最多含 5 个问题（人员/交通/天数划分/儿童年龄/特殊需求），超过 AskUserQuestion 单轮 4 题上限时拆为 2 轮。推荐拆法：第 1 轮放人员+交通+天数划分（条件性），第 2 轮放儿童年龄（条件性）+特殊需求。无条件触发的字段（人员/交通/特殊需求）优先同轮；条件性字段（儿童年龄/天数划分）按是否触发决定是否占位。

1. **人员构成**
   - 选项：1个成人、2个成人、2成人+1儿童、2成人+老人
   - 用户可选择"Other"输入更复杂的人员构成（如单独出行+儿童、多儿童/多老人等）

2. **衣物按人拆卡判定**（零追问，全自动）

   **拆卡条件**：query 含家庭/伴侣称呼词才拆——带爸妈 / 一家 N 口 / 带娃 / 带孩子 / 亲子游 / 和老公 / 和老婆 / 和男/女朋友 / 两口子；或人员构成选项含"老人""儿童"。除此以外（朋友、闺蜜、兄弟、同事、独行、纯人数无称呼）一律不拆，只列"我"的衣物。

   **性别推断**：完全从称呼词解析（爸→M / 妈→F / 老公→M / 老婆→F / 儿子→M / 女儿→F）；推断不出走中性（裤子路线，不列裙子）。**绝不追问**性别 / 关系 / "家人还是朋友"。

3. **交通方式**（multiSelect: true，最多 4 选项）
   - 选项：飞机、自驾、火车/高铁、其他（邮轮/大巴等）
   - 不同交通方式影响携带物品限制和专属装备（见下方交通方式-物品映射）

4. **多城市天数划分**（仅当目的地包含 ≥ 2 个城市，且用户未在原始 query 中显式说明各城天数时**必问**；**若已通过 2.2 文件解析路径完成回显确认，跳过本题**；**若用户在 2.1 选择了「快速版」，跳过本题**）
   - 选项动态生成：基于"总天数 + 城市数"，给出 3-4 种合理划分方案（如悉尼+墨尔本 10 天 → "悉尼 5 + 墨尔本 5"、"悉尼 4 + 墨尔本 6"、"悉尼 6 + 墨尔本 4"，并提供 Other 自定义入口）。
   - 用户回答后，按城市顺序映射到具体日期段（Day 1~N → 城市A，Day N+1~M → 城市B），用于"按天穿搭"分段渲染。
   - 城市顺序若用户未指定，默认按"先国际门户 / 先北后南 / 先大后小"顺序，并在生成时显式声明并允许用户在追问中调整。

5. **儿童年龄**（仅当 Step 4 人员构成中包含儿童时追问，否则跳过）
   - 选项：0-3岁（婴幼儿）、4-7岁（学龄前）、8-12岁（学龄儿童）
   - 不同年龄段携带物品差异大，生成逻辑按年龄差异化处理（见下方儿童年龄-物品映射）

6. **特殊需求**（multiSelect: true，最多 4 选项）
   - **核心原则**：每个选项只描述一个主题，不捆绑无关类别；已被出行类型或人员构成自动覆盖的装备不重复询问。
   - **自动排除规则**（基于出行类型 + 人员构成，已包含的装备不出现为选项）：
     | 出行类型 / 人员构成 | 已自动包含，从选项中排除 |
     |---------------------|------------------------|
     | 摄影之旅 / 摄影/深度游 | 摄影器材（相机/电池/存储卡/三脚架） |
     | 潜水/水上运动 | 水上装备（泳衣、浮潜面镜、防水手机袋、水母衣、潜水袜） |
     | 滑雪 | 滑雪装备（滑雪镜、滑雪手套、护膝护臀、暖宝宝、雪套） |
     | 户外运动（徒步/登山/露营） | 露营装备（帐篷、睡袋、防潮垫、头灯、户外炊具） |

   - **亲子游排除触发条件**：仅当满足下列任一条件时触发——①用户在出行类型中**显式选择或输入**"亲子游"（含 Other 输入"带娃""带小孩"等）；②人员构成中选了含儿童的选项（如"2成人+1儿童"）。仅选"休闲度假"不触发亲子游排除（休闲度假可能是蜜月或纯放松）。

     1. **露营装备**（帐篷、睡袋、防潮垫、头灯、户外炊具）
     2. **宠物随行**（航空箱、牵引绳、食盆水碗、疫苗证明）
     3. **电子设备**（含无人机，触发第 12 节"无人机专项规则"）
     4. **额外医疗用品**（处方药、特殊医疗器械、大量创可贴/绷带等超出基础药品包的物品）
   - **动态组装示例**：
     | 用户选择 | 展示的 4 个选项 |
     |---------|----------------|
     | 亲子游 + 休闲度假 | 露营装备、宠物随行、电子设备、无额外需求 |
     | 潜水/水上运动 | 露营装备、宠物随行、电子设备、无额外需求 |

### 自检表（生成清单前逐项检查）

| 字段 | 通过标准 | 不通过时的处理 |
|------|---------|----------------|
| 出发日期 | 精确到 `YYYY-MM-DD` 或 `M/D`，**或** approximate=true 且已记录锚点日期 | 暂停生成，发起 AskUserQuestion 追问（仅单次；用户选"暂未确定"则自动通过） |
| 行程天数 | > 0 且 ≤ 365，且能推出返回日期；**或** `approximate_days=true` 且 `days=7`（自动通过，不追问） | 仅当二者均不满足时暂停生成、立即追问；用户选过"暂未确定，按 7 天估算"或填过模糊词后自动通过 |
| 各城市天数（多城市行程） | 总和 = 行程天数 | 暂停生成，立即追问划分（快速版豁免：用户选择快速版时此项自动通过） |
| 文件解析回显（仅走文件解析分支时） | 高置信度自动通过；低置信度需用户点击「✅ 信息正确」 | 低置信度时暂停生成，先获得用户确认再调用天气 API——错误的日期/城市会直接污染整张清单 |
| 人员构成 | 已知具体大人/小孩/老人数量 | 走 Step 4 提问 |
| 含儿童时的年龄段 | 0-3 / 4-7 / 8-12 三选一 | 走 Step 4「儿童年龄」提问 |
| 交通方式 | 至少 1 项 | 走 Step 4 提问 |

**自检失败时不要强行继续**：宁可多问一次 AskUserQuestion，也不要用"7 月默认 7/15"、"10 天默认前 5 天 A 城后 5 天 B 城"等隐式假设填补缺口——这类自补默认值一旦和用户真实计划错位，整张清单的衣物件数、按天穿搭、紧急联系都会跟着错位，返工成本远高于多问一次。

## HTML 输出规范

生成一个单文件 HTML，包含所有 CSS 和 JavaScript，无外部依赖。

**生成流程（分离式，提速 ~60%）：**

模板已拆分为三部分（统一存放在 `assets/templates/`）：
- `assets/templates/html-head.html`（542行）：`<!DOCTYPE>` 到 `<div class="container">`，含全部 CSS
- **AI 生成的 body 内容**（~120-200行）：`.container` 内部的所有内容块
- `assets/templates/html-tail.html`（558行）：`</div>` 闭合 container + 全部 JS + `</body></html>`

**AI 只需输出 body 内容区**，写入临时文件（如 `/tmp/body-content.html`），然后执行拼接：
```bash
bash {skill_dir}/scripts/assemble.sh /tmp/body-content.html {output_path}
```

**body 内容区包含的模块**（按顺序）：

## 输出检查清单

> **静默执行**：自检过程不在对话中输出。仅当发现不合规项时向用户说明并修正，合规则直接交付文件。

生成 HTML 后，自检以下 6 项核心业务逻辑（技术校验由 `validate_skill.py` 覆盖）：

- [ ] **目的地与行程完整性**：所有城市出现在概览区；若走文件解析分支，低置信度已获用户确认，高置信度已简要回显
- [ ] **签证状态已验证**：国际出行时，「证件与文件」中签证条目的 label + note 来自 packing-rules §-1「已验证签证表」查表结果或 WebSearch 验证结果；**绝不允许**出现凭推断填写的"免签 X 天"——签证政策按国籍区分，AI 极易混淆不同国籍的免签待遇
- [ ] **info-card vs category-card 边界**：告知类内容（海关申报/入境提醒/禁带物品/紧急电话/法规）只在 info-card，未渲染为 category-card 的可勾选项；category-card 内 item label 不含"⚠/禁/勿/须申报/注意/提醒/警告"等告知性句首
- [ ] **证件文件去重**：国际出行时，护照/签证/机票行程单/酒店确认单/保险单/邀请函等只在「证件与文件」分类出现一次，不重复出现在其他分类；生成完成后跑 packing-rules §6 长清单去重扫描
- [ ] **衣物数量自洽**：item label 中的数量（×N）= ceil(天数 / 频率)，与 note 描述一致（如 note 写"每2天1件"则 N = ceil(天数/2)，不等于天数）
- [ ] **天气驱动穿搭**：天气数据正确映射到按天穿搭建议，温度段与推荐衣物匹配；多城行程若涉及换城日已按 packing-rules §5 追加换城提醒
- [ ] **区域套件 + 气候触发器**：目的地命中 regional-kits.md 时对应装备/证件/info-card 提醒已拼装；info-card 中的政策/电话条目已追加时效性水印；临走待办通过 html-output-rules"类型白名单"过滤，无机票/酒店/攻略等行程规划域条目
- [ ] **validate_skill.py 通过**：运行 `python3 scripts/validate_skill.py` 验证 skill 完整性（33 项技术校验全部通过）

## 错误处理与退出条件

### 异常处理

1. **天气 API 失败**
   - **fallback（回退方案）**：如果 Open-Meteo 请求失败，使用城市的气候平均值作为回退
   - 提示用户："天气数据暂时无法获取，已使用历史平均温度，建议出行前再次确认"

2. **城市名无法识别**
   - 如果 geocoding 返回空结果，要求用户确认城市名（中英文）
   - 提供常见拼写纠正

3. **日期无效**
   - 出发日期不能是过去
   - 天数必须 > 0 且 ≤ 365

4. **参考文件读取失败**
   - **fallback（回退方案）**：若 `packing-rules.md` 或 `references/` 下文件无法读取，使用 SKILL.md 内嵌的概要规则继续生成
   - **escalate（升级处理）**：若内嵌概要也不足以生成合理清单，向用户说明情况并建议重新发起请求

### 退出条件与回滚路径

5. **用户主动退出**
   - 如果用户拒绝提供目的地信息，退出并提示"需要至少一个目的地才能生成清单"
   - 如果天气 API 连续失败 3 次，停止重试，使用 fallback 回退方案（气候平均值）

6. **rollback（回滚）路径**
   - **采集阶段回滚**：若用户在回显确认时选择"🔄 改用问答"或"✏️ 我要改一两项"，回滚到对应的采集步骤重新提问，已确认的字段保持不变
   - **文件解析回滚**：若文件解析连续失败 2 次，自动 rollback 到 Step 2 手动输入路径，并提示"已切换到问答模式"
   - **生成阶段回滚**：若自检表发现字段缺失，rollback 到采集阶段补问，不强行生成不完整清单

7. **escalate（升级）路径**
   - 若用户输入同时涉及多个 skill 的能力范围（如既要行李清单又要酒店预订），escalate 提示用户分别处理或确认优先级
   - 若行程复杂度超出本 skill 处理能力（如 > 10 城市、> 60 天行程），escalate 告知用户建议分段处理

8. **fallback（降级）路径汇总**
   | 异常场景 | fallback 方案 |
   |---------|-------------|
   | 天气 API 不可用 | 使用历史气候平均值 |
   | 参考文件读取失败 | 使用 SKILL.md 内嵌概要规则 |
   | html2canvas CDN 不可用 | 降级为「复制文本」模式 |
   | 文件解析失败 | 回退到手动问答采集 |
   | 多城市天数划分缺失 | 快速版模式（整体温度范围） |
   | 行程天数未定（用户选"按 7 天估算"或填模糊词） | `approximate_days=true`，按 7 天拉天气+计算件数，渲染 `.approx-days-notice` 横幅+两段式穿搭+衣物条目"按 7 天估算"提示，临走待办追加"出发前一周确认实际天数" |
